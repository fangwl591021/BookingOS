# B6.4 Guest Cancellation Token ADR and Migration Proposal

## Status

Proposed. Documentation-only PR. Not implemented. Not deployed.

## Context

BookingOS currently has two customer-facing cancellation paths:

1. Authenticated Customer Session cancellation.
2. Transitional Guest `bookingId + phone` fallback cancellation.

B6.2 blocked valid Customer Sessions with tenant or customer mismatches from falling through to Guest fallback. B6.3 aligned Customer and Guest point rollback order, but Guest fallback still relies on a low-entropy identifier pair that can be guessed or enumerated if exposed poorly.

The long-term guest cancellation model must move to a cancellation token that is generated by the system, stored only as a hash, scoped to tenant and booking, and transmitted only through controlled customer-facing links.

## Decision

New bookings must use Guest cancellation token-only authorization for guest cancellation. New bookings must not rely on `bookingId + phone` for cancellation authorization.

The transitional `bookingId + phone` fallback is allowed only for existing bookings that do not have a cancellation token. It must be removed after a measured backfill and sunset period.

The token system does not solve cancellation transactionality, point rollback atomicity, notification delivery, or persistent idempotency. Those remain separate concerns.

## Security Requirements

### Token Entropy

- Token plaintext must be generated with at least 128 bits of cryptographic randomness.
- Recommended format: 32 bytes random, base64url encoded.
- The database must never store plaintext token values.
- Plaintext may appear only at creation time or controlled resend time.

### Hashing

- Store only `token_hash`.
- Recommended hash input: `tenant_id + ":" + booking_id + ":" + token_plaintext`.
- Recommended hash algorithm: SHA-256 or stronger.
- Compare hashes with constant-time comparison when both values are available in application memory.
- Do not log plaintext token, full hash, or token-bearing URLs.

### Error Anti-Enumeration

Guest token verification failures must return one generic failure response for:

- unknown token
- wrong tenant
- wrong booking
- expired token
- revoked token
- already used token
- non-cancellable booking state

Recommended public message:

```json
{
  "ok": false,
  "error": {
    "code": "CANCELLATION_NOT_AVAILABLE",
    "message": "此預約目前無法取消，請聯絡店家。"
  }
}
```

The server may log safe reason codes internally, but must not expose them to the caller.

## Proposed Table

Use an independent token table instead of adding token columns to `bookings`. This preserves rotation, revocation, audit, and multi-token history without expanding booking row semantics.

Table name proposal: `booking_cancel_tokens`

```sql
CREATE TABLE booking_cancel_tokens (
  id TEXT PRIMARY KEY,
  tenant_id TEXT NOT NULL,
  booking_id TEXT NOT NULL,
  token_hash TEXT NOT NULL,
  status TEXT NOT NULL DEFAULT 'active'
    CHECK (status IN ('active', 'used', 'revoked', 'expired')),
  expires_at TEXT NOT NULL,
  used_at TEXT,
  revoked_at TEXT,
  revoked_reason TEXT,
  replaced_by_token_id TEXT,
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  created_by TEXT,
  created_reason TEXT,
  last_verified_at TEXT,
  verify_count INTEGER NOT NULL DEFAULT 0,
  metadata_json TEXT NOT NULL DEFAULT '{}',
  FOREIGN KEY (tenant_id) REFERENCES tenants(id),
  FOREIGN KEY (booking_id) REFERENCES bookings(id),
  FOREIGN KEY (replaced_by_token_id) REFERENCES booking_cancel_tokens(id)
);
```

## Proposed Unique Keys and Indexes

```sql
CREATE UNIQUE INDEX idx_booking_cancel_tokens_hash
  ON booking_cancel_tokens(token_hash);

CREATE INDEX idx_booking_cancel_tokens_booking_active
  ON booking_cancel_tokens(tenant_id, booking_id, status, expires_at);

CREATE INDEX idx_booking_cancel_tokens_expiry
  ON booking_cancel_tokens(status, expires_at);

CREATE INDEX idx_booking_cancel_tokens_replaced_by
  ON booking_cancel_tokens(replaced_by_token_id);
```

Notes:

- `token_hash` should be globally unique because it is derived from high entropy input and tenant/booking scope.
- Runtime lookup should still confirm `tenant_id`, `booking_id`, `status`, and `expires_at` after hash match.
- Tenant-scoped indexes support audit and cleanup.

## Authorization Priority

Cancellation authorization order should be:

1. Valid Customer Session matching tenant and booking customer.
2. Guest cancellation token matching tenant, booking, active status, and expiry.
3. Transitional legacy `bookingId + phone` fallback only when the booking has no token record and the booking was created before the token rollout cutoff.

A valid but mismatched Customer Session must not fall through to token or phone fallback unless the user explicitly starts a separate guest-token flow without session context. The safer default is fail closed when authenticated identity conflicts with the booking.

## Token Lifecycle

### Generate

When a new guest booking is created:

1. Generate plaintext token with cryptographic randomness.
2. Compute scoped `token_hash`.
3. Insert active token row with tenant and booking scope.
4. Build a cancellation URL containing plaintext token.
5. Send or display the link through the existing booking confirmation channel.

### Transmit

Cancellation URLs should use a token parameter or opaque path segment.

Recommended route shape for B6.5 discussion:

```text
/store/{slug}/cancel?t={token}
```

or

```text
/api/bookings/cancel-token
```

The URL must not include phone as an authorization secret.

### Verify

On token cancellation request:

1. Parse token.
2. Hash token using scoped hash input.
3. Load token row by hash.
4. Validate tenant scope, booking scope, status, expiry, and booking cancellability.
5. Execute cancellation command path.
6. Mark token used or revoked after successful cancellation according to final transaction/idempotency design.

### Expire

A token is invalid after `expires_at`.

Recommended default:

- expires at booking end time plus a short operational grace window, or
- expires at cancellation cutoff time if the product defines one.

Expired token cleanup can be deferred to a scheduled maintenance task.

### Revoke

Revoke token when:

- booking is cancelled by any path
- booking is completed or no_show
- store manually disables guest cancellation
- security rotation is requested

### Rotate

Rotation creates a new active token and marks the old token as `revoked` with `replaced_by_token_id` pointing to the new row.

Plaintext of the new token is only available at rotation time and resend time.

### Cancel Success

After cancellation succeeds:

- mark token `used`
- set `used_at`
- prevent reuse

Whether marking token `used` must be in the same D1 transaction as booking cancellation remains an unresolved B6.5 decision.

## Transitional Phone Fallback

Legacy `bookingId + phone` fallback remains only for existing bookings that meet all conditions:

- booking was created before token rollout cutoff
- no active or historical token row exists for that booking
- booking remains in a cancellable state
- request provides the currently stored customer phone

New bookings created after token rollout must always have a token and must not use phone fallback.

Removal conditions:

1. All active future guest bookings have token rows.
2. Legacy no-token bookings have expired, completed, cancelled, or been backfilled.
3. Metrics show no successful phone fallback cancellations for a defined observation window.
4. Product confirms phone fallback sunset.

## Backfill Plan

Backfill is required for existing future guest bookings that do not have tokens.

Suggested process:

1. Dry-run count by tenant:
   - future bookings
   - cancellable statuses
   - no token row
   - no customer session requirement
2. Generate token rows for eligible bookings.
3. Do not expose plaintext for historical bookings unless a controlled resend action is executed.
4. Mark backfilled rows with `created_reason = 'backfill'`.
5. Add observability for token row coverage percentage.

Backfill must avoid logging plaintext token values.

## Rollout Plan

1. Add schema through a dedicated migration PR.
2. Add write path for token creation on new guest booking.
3. Add token verification read path behind a feature flag or route-level gate.
4. Add customer-facing cancel page or endpoint.
5. Keep phone fallback only for no-token legacy bookings.
6. Add operational metrics.
7. Disable phone fallback after removal conditions are met.

## Rollback Plan

If token rollout fails before phone fallback removal:

- disable token verification route or feature flag
- keep existing Customer Session cancel
- keep legacy phone fallback for no-token bookings
- do not delete token rows
- mark problematic tokens as revoked only when needed

If migration must be rolled back, a rollback SQL proposal should drop indexes first, then the table. This should be used only before production data depends on token rows.

Rollback SQL proposal:

```sql
DROP INDEX IF EXISTS idx_booking_cancel_tokens_replaced_by;
DROP INDEX IF EXISTS idx_booking_cancel_tokens_expiry;
DROP INDEX IF EXISTS idx_booking_cancel_tokens_booking_active;
DROP INDEX IF EXISTS idx_booking_cancel_tokens_hash;
DROP TABLE IF EXISTS booking_cancel_tokens;
```

## Observability

Track counts only. Do not log plaintext token, full hash, phone, LINE UID, or customer PII.

Recommended safe metrics:

- token_created
- token_verified_success
- token_verified_failed_generic
- token_expired
- token_revoked
- token_used
- legacy_phone_fallback_success
- legacy_phone_fallback_rejected
- booking_without_token_count
- cancellation_token_coverage_ratio

Safe diagnostic dimensions:

- tenant_id
- route name
- failure class
- token status bucket
- booking status

## Testing Strategy

### Unit and Contract Tests

- token generation uses high entropy and unique values
- only token hash is persisted
- wrong token returns generic error
- expired token returns generic error
- revoked token returns generic error
- used token returns generic error
- wrong tenant fails closed
- wrong booking fails closed
- token lookup remains tenant scoped
- phone fallback is blocked for new tokenized bookings
- phone fallback remains for eligible legacy no-token bookings

### Integration Tests

- guest booking creation creates token row
- guest token cancellation cancels matching booking
- Customer Session cancellation priority remains first
- mismatched Customer Session does not fall through
- cancellation marks token used or invalidated according to B6.5 transaction design
- cancelled/completed/no_show bookings cannot be cancelled with token

### Migration Tests

- migration applies cleanly
- indexes exist
- backfill dry-run counts are correct
- rollback SQL works only in non-production test database

## Relationship to Existing Risks

Guest cancellation token improves authorization and anti-enumeration. It does not make cancellation atomic.

Existing unresolved risks remain:

- cancellation status update and point rollback are not in a D1 transaction
- notification delivery is not persistent idempotency
- retry/replay stored result is not implemented
- token used/revoked update may need transaction semantics with booking cancellation

These must not be described as solved by B6.4.

## B6.5 Recommended Minimal Scope

B6.5 should be a migration and narrow runtime implementation PR only after this ADR is accepted.

Recommended in scope:

- add `booking_cancel_tokens` migration
- create token rows for new guest bookings
- add token verification helper
- add guest token cancellation route or extend cancel route with explicit token mode
- keep Customer Session cancel unchanged
- keep phone fallback only for eligible no-token legacy bookings
- add tests for token auth, anti-enumeration, and fallback gating

Recommended out of scope:

- full transaction redesign
- persistent idempotency
- notification rewrite
- LINE/Web Push changes
- UI redesign beyond the minimum cancel link flow
- removal of phone fallback before coverage metrics support it

## Unresolved Decisions Before B6.5

1. Exact token expiry policy.
2. Exact public route shape.
3. Whether token `used` marking must be in the same transaction as cancellation.
4. Whether to add a feature flag for token-only guest cancellation rollout.
5. Whether backfilled legacy bookings should receive cancellation link resend automatically or only on demand.
6. Final phone fallback sunset date or metric threshold.
